SECTION 1: EXECUTIVE SUMMARY & REAL-LIFE RELEVANCE
--------------------------------------------------------------------------------
In modern fintech and digital banking (e.g., KYC onboarding, payment processing,
document verification), relying on a single third-party vendor creates single
points of failure, vendor lock-in, and unpredictable costs/latency. 

This platform acts as an intelligent API Gateway. The client application calls
one unified endpoint (/route) without knowing or caring which backend vendor
(Vendor A, Vendor B, Vendor C) processes the request. The routing engine
dynamically analyzes real-time health metrics, rate limits, cost per call,
latency variance, and AI-generated rules to route traffic to the best vendor.

SECTION 3: CORE VALIDATION ANSWERS (FOR ASSIGNMENT EVALUATION)
--------------------------------------------------------------------------------

Q1: How does the client remain agnostic of which vendor is selected?
ANS: The platform exposes a single unified API endpoint (/route). The client
     sends only the required "capability" (e.g., SMS, OCR) and generic payload.
     The routing engine encapsulates vendor selection, payload transformation,
     and vendor-specific authentication. The final response is normalized into
     a standard JSON format so the client code never changes even if vendors
     are added, removed, or swapped.

Q2: How are all 8 routing strategies implemented and validated?
ANS: 1. Priority: Sorts eligible vendors by ascending numerical priority score.
     2. Weighted: Uses weighted random selection based on configured traffic percentage.
     3. Lowest Latency: Evaluates real-time moving average latency signals.
     4. Lowest Cost: Sorts eligible vendors by costPerRequest to optimize budget.
     5. Failover: Iterates through fallback vendors sequentially upon errors/timeouts.
     6. Round Robin: Distributes requests evenly across vendors using cyclic counters.
     7. Feature Based: Filters vendors by matching required document/transaction types.
     8. Health Based: Computes a composite health score ((1 - errorRate) * availability)
        and routes traffic to the most reliable vendor.
     All 8 are validated in unit tests (tests/strategies.test.js) and seeded logs.

Q3: How does the system handle vendor timeouts, rate limits, and failures?
ANS: - Rate Limiting: An in-memory sliding window rate limiter tracks vendor API
       calls per minute. If a vendor reaches its limit, it is temporarily marked
       unavailable and traffic routes to the next best vendor.
     - Timeouts: Each vendor configuration has a defined timeoutMs. If an HTTP
       call exceeds this threshold, the request is aborted and automatically
       handed off to the failover chain.
     - Circuit Breaking / Health: Vendors with repeated failures are marked
       'degraded' or 'down', preventing black-hole traffic routing.

Q4: How is request tracking and auditing handled?
ANS: Every routing request generates a standard RFC 4122 UUID (e.g., c82f9d1a-...).
     This UUID is stored in MongoDB along with the full payload, selected vendor,
     evaluation reason, total latency, and complete failover history. In the
     UI, IDs are displayed as clean hex badges (#c82f9d1a) with tooltip support
     for full traceability without cluttering the screen.

Q5: What is the purpose of the AI Rule Generator and Strategy Advisor?
ANS: - AI Rule Generator: Allows business operators without coding skills to
       input natural language policies (e.g., "Route VIP customers to Vendor A"),
       which are compiled into executable JSON AST rules.
     - Strategy Advisor: Continually analyzes vendor performance drift (e.g.,
       sudden latency spikes or cost increases) and recommends strategy changes
       to system administrators.

Q6: What is the use of the "Save & Apply to Vendors" button on the AI Rule Generator tab?
ANS: The "Save & Apply to Vendors" button bridges the gap between AI generation
     and live backend execution:
     1. Persists Rule in MongoDB: Saves the generated JSON routing rule into the
        database (RoutingConfig collection) with isActive: true.
     2. Modifies Live Vendor Database:
        - If the rule specifies percentage splits, it automatically updates the
          real 'weight' property of those vendors in MongoDB.
        - If the rule specifies a failover order, it automatically updates the
          'priority' scores of those vendors in MongoDB.
     3. Instant Cache Invalidation: Flushes the server's in-memory vendor cache
        so the next routing request instantly executes using the new policy!

Q7: Does the platform route requests automatically without manually selecting a rule?
ANS: YES! In real-life production API usage, external client applications calling
     POST /route never manually select an AI rule from a dropdown menu.
     - Automatic Production Behavior: When an API client calls /route with only
       { "capability": "PAN", "payload": { ... } }, the backend routing engine
       automatically queries MongoDB for the currently active policy (isActive: true),
       and automatically adopts its strategy and ranking conditions!
     - Why Route Tester has a dropdown: The dropdown on the Route Tester tab is
       strictly a "Sandbox Override" tool for administrators to test draft rules
       before making them active in production. By default, it is set to
       "✨ Auto-Route using Active AI Policy (Default)".

Q8: How does the dashboard receive real-time telemetry without server polling?
ANS: To eliminate continuous HTTP polling and prevent unnecessary server load,
     the system uses Unidirectional WebSocket Push architecture:
     - Server-Side Socket Service (server/services/socketService.js): Attaches a
       WebSocket server on /ws. When a routing decision is made (/route) or when
       vendors are updated (CRUD), the server broadcasts lightweight push events
       (LOG_UPDATE, TELEMETRY_REFRESH, VENDORS_UPDATE).
     - Client Listener (client/src/services/socket.js): All frontend hooks
       (useHealth, useMetrics, useLogs, AppContext) subscribe to these WebSocket
       events. Data refreshes instantly ONLY when real backend events occur,
       with zero continuous polling overhead!
